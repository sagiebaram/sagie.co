2. Create `vitest.config.ts` at project root:
   ```typescript
   import { defineConfig } from 'vitest/config'
   import path from 'path'

   export default defineConfig({
     resolve: {
       alias: {
         '@': path.resolve(__dirname, './src'),
       },
     },
     test: {
       environment: 'node',
       globals: true,
       coverage: {
         provider: 'v8',
         reporter: ['text', 'json'],
         include: ['src/lib/**/*.ts'],
         exclude: [

           'src/lib/gsap.ts',
         ],
       },
     },
   })
   ```

<task type="auto" tdd="true">
  <name>Task 2: Unit tests for Zod schemas and withValidation middleware</name>
  <files>src/lib/__tests__/schemas.test.ts, src/lib/__tests__/validation.test.ts</files>
  <behavior>
    schemas.test.ts:
    - MembershipSchema: accepts valid {fullName, email, role, location}, rejects missing email, rejects invalid email format
    - ChapterSchema: accepts valid input, rejects missing required fields
    - VenturesSchema: accepts valid input, rejects missing required fields
    - SolutionsSchema: accepts valid input, rejects missing required fields
    - Each schema: trims string fields, lowercases email

    validation.test.ts:
    - Honeypot filled (_trap non-empty) -> 200 silently, handler NOT called
    - Timing too fast (_t within 3000ms of Date.now) -> 200 silently, handler NOT called
    - Invalid JSON body -> 400
    - Zod validation failure (missing required fields) -> 422 with fieldErrors
    - Valid body with sufficient elapsed time -> calls handler, returns handler response
    - Rate limiting: 6th request from same IP -> 429 with Retry-After header
  </behavior>
  <action>
1. Create `src/lib/__tests__/schemas.test.ts`:
   - Import all schemas from `@/lib/schemas`
   - Mock `server-only` as empty module: `vi.mock('server-only', () => ({}))`
   - For each schema (MembershipSchema, ChapterSchema, VenturesSchema, SolutionsSchema):
     - Test `.safeParse()` with a fully valid input returns `success: true`
     - Test `.safeParse()` with missing required fields returns `success: false`
     - Test `.safeParse()` with invalid field types (e.g., number for email) returns `success: false`
     - Test string trimming: input with leading/trailing whitespace is trimmed in output
     - Test email lowercase: uppercase email is lowercased in output
   - Use `describe` blocks per schema for organization
   - No mocking of notion or env needed — schemas are pure Zod, only import from `zod`

2. Create `src/lib/__tests__/validation.test.ts`:
   - Mock `server-only`: `vi.mock('server-only', () => ({}))`
   - Mock `@/env/server`: provide `env.ALLOWED_ORIGINS` = `'http://localhost:3000'`
   - Import `withValidation` from `@/lib/validation`
   - Import `MembershipSchema` from `@/lib/schemas`
   - Create helper `makeRequest(body, extraHeaders?)` that builds a `new Request(...)` with JSON body
   - Mock `Date.now` with `vi.spyOn(Date, 'now')` for timing tests — avoids flakiness
   - Create `const mockHandler = vi.fn().mockResolvedValue(Response.json({ success: true }))`
   - Construct handler: `const handler = withValidation(MembershipSchema, mockHandler)`
   - Test cases:
     a. Honeypot: `_trap: 'bot'` in body -> res.status === 200, mockHandler not called
     b. Fast timing: `_t` set to `Date.now() - 100` (less than 3000ms) -> res.status === 200, handler not called
     c. Invalid JSON: body is `'not-json'` string -> res.status === 400
     d. Zod failure: valid timing, missing required fields -> res.status === 422, body has `fieldErrors`
     e. Valid request: all fields present, `_t` set to `Date.now() - 5000` -> mockHandler called once
     f. Rate limit: call handler 6 times from same IP -> 6th returns 429 with `Retry-After` header
   - Use `beforeEach` to clear mocks and reset rate limiter state (note: rate limiter uses in-memory Map in validation.ts — tests may need to account for accumulated state across tests, so run rate limit test last or use a fresh handler per test)

   IMPORTANT for rate limiter tests: The rate limiter is module-level state in validation.ts. Since vi.mock does NOT reset module state between tests, the rate limit Map accumulates. Either:
   - Run rate limit tests in a dedicated describe block at the end
   - Or accept that previous test calls count toward the limit and adjust the test accordingly
   - The simplest approach: create a separate `withValidation` handler instance for the rate limit test block, and make requests with a unique IP header (`x-forwarded-for`) that no other test uses
  </action>
